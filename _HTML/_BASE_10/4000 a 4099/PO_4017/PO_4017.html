• ESPECIFICAR VALIDAÇÃO DO ESQUEMA JSON

Esquema JSON é um vocabulário que permite anotar e validar documentos JSON. Você pode usar o esquema JSON para especificar regras de validação para seus campos em um formato legível.

Neste exemplo, você cria uma studentscoleção com regras de validação e observa os resultados depois de tentar inserir um documento inválido.

Crie uma coleção com validação. Em mongosh, execute o seguinte comando para criar uma studentscoleção e usar o $jsonSchemaoperador para definir regras de validação de esquema:

  • 1. validator - É uma funcionalidade usada para validar os dados inseridos ou atualizados em uma coleção, garantindo que eles sigam um determinado esquema ou regras específicas. Ele é definido no momento da criação da coleção ou pode ser adicionado posteriormente.
    • 1.1 - Impor regras de integridade nos documentos.
    • 1.2 - Evitar dados inconsistentes ou malformados.
    • 1.3 - Controlar o formato dos dados (por exemplo, tipos, valores obrigatórios, padrões de string, etc.).
    • Níveis de validação: Você também pode definir o nível de severidade da validação
      • 1.4 - strict: rejeita documentos que não seguem o esquema.
      • 1.5 - moderate: permite documentos inválidos em algumas operações (como atualizações internas).
  • 2. $jsonSchema - é uma forma de definir regras de validação de documentos no MongoDB usando a especificação JSON Schema, que é um padrão para descrever a estrutura e o conteúdo de dados JSON. Ele é usado dentro do campo validator para garantir que os documentos inseridos ou atualizados em uma coleção sigam um formato específico.
    • 2.1 - bsonType: Define o tipo de dado esperado (string, int, object, array, etc.).
    • 2.2 - required: Lista de campos obrigatórios.
    • 2.3 - properties: Define as regras para cada campo.
    • 2.4 - minimum / maximum: Para valores numéricos.
    • 2.5 - pattern: Expressão regular para validar strings (como e-mails, por exemplo).
    • 2.6 - enum: Lista de valores permitidos.
      • Vantagens de usar $jsonSchema:
        • Validação mais robusta e legível.
        • Compatível com ferramentas que já usam JSON Schema.
        • Permite validações complexas sem precisar de lógica na aplicação.
  • 3. bsonTypeobect - é usado dentro de um $jsonSchema para indicar que o valor de um campo deve ser um objeto BSON, ou seja, uma estrutura de chave-valor aninhada, semelhante a um objeto JSON.
    • 3.1 O que é um object no BSON? - Um campo com bsonType: "object" representa um documento embutido (ou subdocumento). Isso é útil quando você quer validar estruturas internas dentro de um documento principal.
    • 3.2 Quando usar bsonType: "object"? - Quando você tem subdocumentos ou estruturas aninhadas. Quando precisa validar campos internos de um objeto. Para manter a organização e integridade dos dados complexos.
    • Neste exemploO campo endereco é um objeto e dentro dele, há validação para os campos rua, cidade e cep.
  • 4. requered o campo required é usado para especificar quais campos são obrigatórios em um documento e recebe uma lista de nomes de campos que devem obrigatoriamente estar presentes no documento. Se algum desses campos estiver ausente durante uma inserção (insert) ou atualização (update), o MongoDB rejeita a operação.
  • Neste exemplo, podemos ver que os campos nome e email são obrigatórios, mas o campo idade é opcional.
  • 5. properties É onde você define as regras de validação específicas para cada campo do documento, sendo um objeto que contém uma entrada para cada campo que você deseja validar, assim Cada entrada define as características esperadas daquele campo, como tipo (bsonType), valores mínimos/máximos, padrões de string, etc.
  • Neste exemplo, properties define as regras para os campos nome, idade e email e Cada campo tem seu próprio conjunto de validações. servindo assim para controlar o formato e os valores de cada campo, garantir que os dados estejam estruturados corretamente, permitir validações específicas por campo.
DICA
Esclareça as regras com campos de título e descrição - Você pode usar os campos titlee descriptionpara fornecer uma explicação das regras de validação quando elas não forem imediatamente claras. Quando um documento falha na validação, o MongoDB inclui esses campos na saída de erro.
